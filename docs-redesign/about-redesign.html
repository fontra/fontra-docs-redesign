About this redesign

This prototype proposes an update to docs.fontra.xyz in two dimensions: shape (information architecture and visual design) and content (what's written, and what's missing). It's a draft for discussion, not a finished site.

What stays the same

The current docs already use the Diátaxis system (tutorials / how-tos / reference / explanations), are open-source and CC BY. That foundation is sound. This redesign keeps Diátaxis and improves the layer on top of it.

Shape — what changed in the structure

  1. A real landing page with three doors. Instead of a single generic entry, the home routes the three audiences that actually arrive: beginners, migrants from other editors, and people building/automating. One door each.
  2. Persistent sidebar + on-this-page TOC. The whole map is visible at all times; long pages get a scroll-spying table of contents. You always know where you are.
  3. Diátaxis, relabeled for humans. "Learn / Guides / Reference / Concepts" reads more plainly than "Tutorials / How-Tos / Reference / Explanations" while keeping the same discipline behind the scenes.
  4. Two new top-level sections: Workflow (the CLI pipeline, currently undocumented) and Migrate (Rosetta-stone pages). Both serve audiences the current site doesn't.
  5. Video-first pages. The project already produces great short clips; here every how-to and tutorial opens with one rather than burying the links at the bottom.
  6. Search. A client-side index (Pagefind/Lunr) — no server, works on GitHub Pages. (Mocked here; press ⌘K.)
  7. Light/dark, responsive, accessible. Theme toggle, mobile drawer nav, keyboard-navigable, system-font stack for speed.

Content — what changed in the writing

  1. A true zero-to-font tutorial. The current Quickstart assumes you already have a UFO. Your first variable font starts from nothing and ends with an installed font in 30 minutes — the single highest-impact page for new users.
  2. Migration pages. Glyphs and RoboFont users need a vocabulary map and a "try it without leaving your tool" path. Churn happens on session two, from habit friction; these pages remove it.
  3. The Workflow cookbook. Recipes first, generated reference second — because people adopt CLI tools by modifying examples.
  4. A few new concept pages (interpolation & compatibility, how Fontra is built, shaping) to give reference pages somewhere to link "why" without bloating.

Generated reference (proposal)

Fontra's Workflow actions are Python dataclasses; their arguments, types and defaults are already machine-readable. The same is true of the menus and panels to a large extent. A build step could generate the most mechanical reference tables from source, so they never drift out of date — and double as autocomplete data for YAML editing. The hand-written prose explains; the generated tables enumerate.

How it would be built

Keep it boring and contributor-friendly: a static-site generator (the current site uses Jekyll; 11ty or Astro are also fine), Markdown content, a components layer for the callouts/video/step patterns shown here, Pagefind for search, and GitHub Pages for hosting — exactly the current deployment model. Nothing here requires a server or a framework rewrite.

⚠ Prototype caveats

Links marked planned or pointing to # are illustrative stubs. Screenshots are represented by striped 🎬 video placeholders. Copy is drafted and would need a review pass against the current app before publishing.
